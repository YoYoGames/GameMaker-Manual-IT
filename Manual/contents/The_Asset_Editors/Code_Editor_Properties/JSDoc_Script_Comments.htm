JSDoc Script Commenti

Se si desidera che le funzioni personalizzate abbiano il completamento del codice e mostrino gli argomenti richiesti in modo specifico nell'editor di codice, è necessario aggiungere alcuni commenti in stile JSDoc. Questi commenti sono usati per indicare alla funzione di completamento automatico come la funzione deve essere usata e compilata nell'editor di script.

Il formato dell'intestazione di una funzione tipica è il nome della funzione, la descrizione della funzione e l'elenco dei diversi argomenti (parametri) che la funzione accetta, assicurandosi di iniziare ogni riga con una tripla barra rovesciata "///", che indica a GameMaker di analizzare il commento come se fosse in stile JSDoc. In alternativa, è possibile avvolgere i commenti JSDoc in /**... */ con /** sulla propria riga.

I commenti stessi devono avere un identificatore (preceduto da "@") e un contenuto; gli identificatori disponibili sono i seguenti:

Identificatore Contenuto Sintassi
@function / @func Il nome completo della funzione, compresi gli argomenti @func my_func(args)
@description / @desc Una descrizione generale di ciò che fa la funzione @desc <Description goes>
@param / @parameter / @arg / @argument Informazioni sugli argomenti, con un'opzione type @param {type} name <Parameter description>
@return / @returns Tipo di dati restituiti dalla funzione @return {type} <Return description>
Solopiume
@url URL for the function, opens when you press F1 on the function name or use the RMB -> Go To Declaration option. @url https://...
@pure Contrassegna la funzione come pura @pure
@ignore Nasconde la funzione dal completamento automatico di Feather @ignore
@deprecated Segna la funzione come deprecata @deprecated
@context / @self Imposta il contesto della funzione, che Feather utilizza per fornire informazioni contestuali. Può essere un object o un costruttore. @self <object>
Legacy Code Editor only
@function / @func The full function name including arguments @func my_func(args)

Se non si è specificato un tipo per gli argomenti o il valore di ritorno e si utilizza Feather, quest'ultimo assumerà automaticamente un data type per essi, in base al corpo della funzione.

Vedere le informazioni su data types per @param e @return sui tipi di dati Feather.

Esempi

Per avere un'idea di come funzionerebbe quando si scrivono le proprie funzioni, prendiamo questo esempio di base:

function is_same_object(_id, _obj)
{
    if (_id.object_index == _obj)
    {
        return true;
    }
    else return false;
}

Tutto ciò che fa questo script è controllare se un'istanza ha lo stesso object_index di un dato object e sarebbe chiamato semplicemente come:

if is_same_object(id, obj_Player)
{
    instance_destroy()
}

Tuttavia, scrivendolo nell'editor di codice si vedranno direttamente i nomi delle variabili degli argomenti (_id e _obj), che nella maggior parte dei casi non sono molto descrittivi. È possibile utilizzare JSDoc per definire nomi e descrizioni degli argomenti personalizzati, insieme alle informazioni per la funzione:

/// @function                is_same_object(inst_id, object_id)
/// @description             Check if the given instance belongs to the given object.
/// @param {Id.Instance}     inst_id    The unique instance ID value of the instance to check.
/// @param {Asset.GMObject}  object_id  The object index to be checked against.
/// @return {Bool}

function is_same_object(_inst_id, _object_id)
{
    return _inst_id.object_index == _object_id;
}

Ora, quando si richiama questa funzione in qualsiasi punto del progetto, si vedranno i nuovi nomi degli argomenti inseriti nei commenti di JSDoc:

Nell'immagine qui sopra, la parte superiore mostra la funzione nel completamento automatico e la parte inferiore mostra come funziona l'helper degli argomenti in basso. Si noti che sia la parte opzionale "type" (tipo) che quella obbligatoria "description" (descrizione) di @param non sono visualizzate per impostazione predefinita nel codice completo di IDE; per vederle è necessario attivare le opzioni nelle Preferenze GML.

Quando si utilizza Feather, si visualizzano informazioni dettagliate sulla funzione quando si passa il mouse su di essa:

È possibile racchiudere il nome di un argomento tra le parentesi [] per indicare che è opzionale. L'editor di codice si aspetterà quindi un numero di argomenti compreso tra il minimo richiesto e il numero totale di argomenti.

Ad esempio, si veda la seguente funzione:

/// @function    animate_position(end_x, end_y, start_x, start_y)
/// @desc        Animates the instance to ending point, from optional starting point
/// @arg end_x
/// @arg end_y
/// @arg [start_x]
/// @arg [start_y]

function animate_position (x1, y1, x2, y2)
{
    // Function code
}

Gli argomenti start_x e start_y sono contrassegnati come opzionali, il che significa che l'editor di codice si aspetta da 2 a 4 argomenti, come si può vedere nel messaggio di avviso:

NOTA Lo stesso comportamento si ottiene se si utilizzano argomenti opzionali nella dichiarazione della funzione. Per ulteriori informazioni, vedere Funzioni di script.
