Parametri della riga di comando

Quando si creano giochi è importante testare il file eseguibile per assicurarsi che la compilazione finale sia corretta e non presenti errori o differenze impreviste. Tuttavia, creare un file eseguibile significa che, non avendo accesso alla console di debug né alla finestra del compilatore, può essere difficile individuare ed eseguire il debug di qualsiasi problema riscontrato. Esiste tuttavia una soluzione, che consiste nell'utilizzare i parametri della riga di comando per impostare alcune proprietà quando viene eseguita la versione eseguibile del gioco, consentendo così di creare file di log di output e di testare altre situazioni speciali.

NOTA Questa funzione è disponibile solo per il modulo target Windows standard.

Per farlo funzionare, è necessario creare un eseguibile Windows del gioco e installarlo sul computer di prova. Una volta fatto ciò, è necessario creare un collegamento al gioco sul desktop. È qui che verranno aggiunti i parametri da passare a *.exe (per creare un collegamento, fare clic con il pulsante destro del mouse su l'eseguibile del gioco e selezionare Invia a > Desktop (crea collegamento)).

Una volta creata la scorciatoia, è possibile utilizzare uno dei seguenti parametri nel percorso di destinazione:

-noaudio: Disattiva tutto l'audio del gioco.

-inawindow: Costringe il gioco ad avviarsi in una finestra, anche quando è impostato per essere eseguito in modalità a schermo intero.

-output <filename>: Invia l'output della console al nome del file, compresi i messaggi di debug presenti nella base di codice.

-debugoutput <filename>: Invia l'output della console al nome del file, escludendo qualsiasi messaggio di debug personalizzato, ma includendo informazioni extra dal runner per la segnalazione di bug.

-software: Forza il gioco a usare il Software Vertex Processing invece dell'hardware per il rendering della grafica del gioco. Questo è particolarmente utile per i PC che utilizzano un chipset Intel GFX integrato o che presentano problemi di visualizzazione su macchine più vecchie.

-intel: Questo switch risolve il problema dei driver Intel difettosi, indipendentemente dalla GPU utilizzata. Questa soluzione per i driver difettosi rallenterà il gioco, causando un notevole calo delle prestazioni, quindi se non avete bisogno della correzione non usatela.

-vanillaGFX: L'uso di questa opzione disattiva qualsiasi controllo del produttore del driver e può essere molto utile per testare il gioco "così com'è".

Vale la pena notare che <filename> in -output e -debugoutput può essere lo stesso, se lo si desidera, poiché il file viene aperto e chiuso a ogni scrittura.

A questo punto, si dovrebbe andare al punto in cui è stato creato il collegamento all'eseguibile del gioco, quindi fare clic con il pulsante destro del mouse su di esso e selezionare Proprietà.

Si aprirà una finestra simile a quella riportata sopra. Qui è possibile aggiungere alla casella di destinazione path la propria riga di comando. Basta scorrere fino alla fine del dato path, aggiungere uno spazio e poi il parametro string. Quindi, ad esempio, per creare un file di debug in uscita, si dovrebbe avere qualcosa di simile a:

D:\Users\Me\GameMaker\CustomLoadBar-Default-1.0.0.0\CustomLoadBar.exe -debugoutput debug.txt

A questo punto verrà creato un file chiamato "debug.txt" nella stessa posizione dell'eseguibile (non nella posizione del collegamento, ma dove è installato il gioco) con le informazioni di debug e di compilazione del gioco finito. Fate clic su "OK" e poi utilizzate il collegamento per eseguire il gioco, in modo che utilizzi questi parametri e aiuti il vostro debugging.

Per ulteriori informazioni su come eseguire il debug del gioco, consultare la sezione del manuale dedicata al debug.

Test A/B con la riga di comando

Il runner YoYo beneficia anche di uno speciale parametro della riga di comando:

-game <filename>: carica il file di gioco indicato

Perché si dovrebbe usare questo? Normalmente non lo fareste, ma se desiderate avere più di una versione del gioco in esecuzione alla volta, potete farlo con questo metodo, rendendo i test comparativi A/B molto più semplici. Con questo metodo è possibile effettuare test mirati con qualcun altro o, in qualità di progettisti, farsi un'idea più precisa delle cose facendo test affiancati.

Per configurare questo sistema, è necessario conoscere un paio di cose in anticipo. Per cominciare, è necessario il sito path del Runner stesso. Questo si trova nella cartella Program Data del sistema operativo Windows (potrebbe essere nascosto per impostazione predefinita, quindi è necessario modificare le proprietà di visualizzazione per vederlo), con un formato tipico come questo:

C:\ProgramData\GameMaker\Cache\runtimes\<runtime_version>\windows\Runner.exe

Si noti che la cartella runtime_version dipenderà dal file runtimes attualmente installato. È necessario anche il file da path a <yourgame>.win, perché è a questo che si vuole far puntare il runner. Il modo più semplice per trovarlo è andare nella cartella Temp (di solito si trova in %localappdata%\Temp\GameMakerStudio\<game_name>\) e cercare *.win, che mostrerà tutti quelli disponibili e si potrà scegliere quello da cui si vuole ottenere il file path completo.

Ora, potremmo creare un collegamento al runner come abbiamo fatto per l'eseguibile e poi modificare i parametri secondo le necessità, ma ciò significa doverlo aprire e modificare tra un test e l'altro, vanificando lo scopo di utilizzare questa funzionalità per i test affiancati. Creeremo invece un paio di piccoli file "batch" per gestire questo aspetto.

A tale scopo, creare un nuovo documento di testo sul desktop e aprirlo con Notepad (o qualsiasi altro editor di testo). Aggiungere quanto segue:

START C:\ProgramData\GameMaker\Cache\runtimes\<runtime_version>\windows\Runner.exe -game <LOCAL TEMP PATH>\<GAMENAME>.win

È necessario impostare il temp path in modo che punti al file *.win del gioco e poi salvarlo come "GM_Test_1.bat" sul desktop (si tenga presente che Notepad salverà automaticamente come file *.txt a meno che non si scelga "tutti i file" dalle opzioni di salvataggio). Un esempio di come dovrebbe apparire il comando finale è:

START C:\ProgramData\GameMaker\Cache\runtimes\runtime-9.9.1.199\windows\Runner.exe -game D:\Users\Mark\AppData\Local\Temp\GameMaker\AlphaDog_Reboot\AlphaDog_Reboot.win

Ora create un altro file di testo e aggiungete la stessa riga, solo che questa volta puntate al file path per la seconda versione del gioco che volete testare fianco a fianco, e salvate questo file come "GM_Test_2.bat". È possibile fare doppio clic su uno dei due per visualizzare le diverse versioni del gioco, e il fatto di averli come file batch significa che è possibile riaprirli facilmente per modificarli e cambiare il gioco da testare.
